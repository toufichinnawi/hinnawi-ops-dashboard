# Teams Webhook Research

## Key Findings

### Modern Approach: Workflows (recommended)
- Microsoft 365 Connectors are being deprecated
- New approach uses Power Automate "Workflows" with "When a Teams webhook request is received" trigger
- Workflows support **Adaptive Cards only** (not older message card format)
- Webhook URL is generated by the workflow

### Adaptive Card JSON Format for Workflows
```json
{
  "type": "message",
  "attachments": [
    {
      "contentType": "application/vnd.microsoft.card.adaptive",
      "contentUrl": null,
      "content": {
        "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
        "type": "AdaptiveCard",
        "version": "1.4",
        "body": [
          {
            "type": "TextBlock",
            "text": "Alert Title",
            "weight": "Bolder",
            "size": "Medium"
          },
          {
            "type": "TextBlock",
            "text": "Alert description here",
            "wrap": true
          }
        ]
      }
    }
  ]
}
```

### Rate Limits
- Max 4 requests per second
- Message size limit: 28 KB
- Use retry logic with exponential backoff

### Implementation Plan
Since this is a static frontend project, we need to upgrade to full-stack to:
1. Store webhook URL securely on the server
2. Send POST requests to Teams from the backend (avoid CORS issues)
3. Evaluate alert rules server-side

### Alternative: Client-side approach
- We could implement the webhook calls from the frontend
- BUT: webhook URLs should be kept secret (anyone with the URL can post to the channel)
- CORS may block direct browser-to-Teams webhook calls
- Better to proxy through our backend

### Decision: Upgrade to web-db-user
- Need backend routes for webhook management
- Need database to store alert rules and webhook URLs
- Need server-side alert evaluation
